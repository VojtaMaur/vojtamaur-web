# Strategie přežití tvého statického webu

## Východisko

Tvůj web je na archivaci a replikaci překvapivě dobře připravený už teď. Architektura je statická, bez CMS a databáze, zdrojem pravdy je repozitář s obsahem a build pipeline, a vedle běžného webového buildu máš i přenositelný souborový build, který přepisuje root-relative cesty tak, aby fungoval mimo klasický serverový root. To je přesně ten typ konstrukce, který se dá kopírovat, balit a ukládat na více nezávislých místech bez toho, aby se při každém přesunu rozpadl. fileciteturn0file0

Důležitý závěr je ale tvrdší: nearchivuješ „web“ jako jednu věc. Archivuješ nejméně tři odlišné artefakty s odlišným profilem rizika. První je zdrojový repozitář s historií a MDX, druhý je zmražený snapshot zdrojů pro reprodukci, třetí je finální přenositelný HTML build pro čtení a obnovu bez toolchainu. Kdo tohle smíchá dohromady, končí se zmatečnou polomrtvou kopií, která ani není dobře čitelná, ani dobře obnovitelná. fileciteturn0file0

## Co má nejvyšší návratnost

Nejlepší základ je pořád veřejný GitHub repozitář, ale ne kvůli romantice open source. Kvůli kombinaci tří věcí: veřejná historie, snadná replikace a release assets. GitHub v běžném Git repu varuje nad 50 MiB, blokuje soubory větší než 100 MiB a doporučuje držet repozitáře ideálně pod 1 GB; zároveň ale release assets mohou mít až 2 GiB na soubor a release jako celek nemá limit celkové velikosti. Pro tvoje poměry z toho plyne jednoduchá věc: zdrojáky a MDX drž v repu, ale offline snapshot webu publikuj jako ZIP/TAR.GZ release asset, ne jako další nafukující vrstvu git historie. GitHub Pages má navíc doporučený limit zdrojového repa 1 GB a publikovaný web do 1 GB, takže čistě z velikosti se tam tvůj build vejde. citeturn8view3turn6view1turn6view0

Jenže GitHub není archiv, je to platforma podmíněná účtem a smluvními pravidly. Jeho vlastní dokumentace přímo říká, že GitHub Pages je vázaný na Terms of Service a že web lze unpublishnout bez mazání obsahu repozitáře. To není detail; to je připomínka, že Pages je distribuční vrstva, ne trvalý depozit. GitHub tedy ber jako primární veřejný forge a rozcestník, ne jako jedinou schránku poslední záchrany. citeturn6view0turn0search0turn9view3

Druhá a třetí kovadlina mají být veřejné mirror forges, ideálně **entity["organization","GitLab","devops platform"]** a **entity["organization","Codeberg","forgejo hosting service"]**. GitLab Pages umí publikovat statické weby přímo z repozitáře, podporuje plain HTML/CSS/JS, běží na infrastruktuře GitLabu bez dodatečných nákladů a na GitLab.com je veřejně dostupný internetově. Codeberg Pages umí totéž přes `codeberg.page`, a navíc stojí na svobodném softwaru `git-pages`/Forgejo, což je důležité z jiného důvodu: když služba jednou chcípne, neumře s ní zároveň i publikační mechanismus. To je kvalitativně lepší profil než čistě proprietární host. citeturn7view1turn7view0turn0search6

Pro samotný zdrojový kód a historii je skoro povinná ještě jedna vrstva: **entity["organization","Software Heritage","source code archive"]**. Jejich „Save Code Now“ veřejně podporuje Git, Mercurial a SVN repozitáře, známé forgy jako GitHub a GitLab odbavuje bez schvalování a dokumentace přímo popisuje jak ruční save request, tak automatické spouštění archivace. Tohle je přesně nástroj na to, aby tvůj zdroj neumřel jen proto, že konkrétní forge jednou změní pravidla nebo účet zmizí. Nevýhoda je evidentní: je to primárně archiv zdrojů, ne hotového HTML buildu. Proto je to vrstva pro MDX + repo, ne náhrada snapshotů webu. citeturn6view3turn9view4

Pro zmražené snapshoty offline buildu je nejsilnější kandidát **entity["organization","Zenodo","research repository"]**. Zenodo podporuje až 50 GB a 100 souborů na záznam, u většího počtu souborů samo doporučuje ZIP archiv a garantuje bit-level preservation, i když negarantuje, že za 10 nebo 50 let bude každý formát stejně pohodlně čitelný. To je pro tebe dobrá zpráva: HTML, ZIP, TAR.GZ, TXT a checksumy jsou přesně ty „nudné“ formáty, které mají archivní smysl. U software release workflow navíc Zenodo dokumentuje GitHub integraci. citeturn8view4turn6view2turn1search12turn1search7

Na artefakty určené ke stažení dává dál smysl i Internet Archive, ale ne tak, jak si to většina lidí naivně představuje. Internet Archive umožňuje běžnému účtu nahrát soubory a metadata jako samostatný item, takže tam můžeš uložit `site-offline.zip`, případně i `source.tar.gz`, `SHA256SUMS` a popis. To je dobré právě proto, že nejde jen o dump dat, ale i o veřejnou katalogizovanou položku. Jen nesmíš zaměnit item upload za Wayback Machine. Wayback „Save Page Now“ ukládá jednu konkrétní stránku, ne celé adresáře ani celé weby, a negarantuje budoucí crawly. citeturn9view0turn9view1turn6view6

Pro český web máš ještě jednu institucionální vrstvu, kterou by byla škoda ignorovat: **entity["organization","Webarchiv","czech web archive"]** provozovaný **entity["organization","Národní knihovna České republiky","national library"]**. Webarchiv dělá plošnou archivaci české domény, výběrovou archivaci podle kritérií a tematické archivace; přijímá návrhy webů a kritéria ti vyhovují téměř ukázkově, protože stačí český původ, čeština, český autor nebo vztah k ČR. To není komerční platforma a není to ani „ještě jeden startup storage“. Je to paměťová instituce s přímo definovaným účelem dlouhodobé ochrany a zpřístupnění českých webových zdrojů. citeturn6view4turn9view2turn3search0

## Co je dobré jako doplněk, ne jako základ

Živá zrcadla mají smysl hlavně kvůli dohledatelnosti a sekundárnímu přepisu do dalších archivních vrstev. Jinými slovy: nejsou to trezory, jsou to návnady pro crawlery, indexy a link graph. U tebe to dává smysl právě proto, že máš hotový statický build. GitLab Pages a Codeberg Pages jsou použitelné pro další veřejné URL bez vlastní infrastruktury, a **entity["company","Cloudflare","web infrastructure company"]** Pages podporuje přímý upload předem sestavených assetů z lokálního počítače nebo vlastního build systému. To je užitečné jako další veřejný povrch, ale pořád je to aktivní hosting vázaný na účet a službu, ne archivní depozit. citeturn7view1turn7view0turn7view2

Tohle má přímou vazbu na Internet Archive i na obecné webové datasety. Wayback Site Search je podle oficiální nápovědy rankovaný mimo jiné počtem capture a počtem relevantních odkazů na homepage. Jinak řečeno: tvoje intuice o „propojení do grafu internetu“ není ezoterika, ale praktická mechanika dohledatelnosti. Čím víc nezávislých veřejných vstupních bodů a odkazů mezi nimi, tím větší šance, že se kolem nich nabalí sekundární kopie a záznamy. citeturn6view6

Common Crawl do toho zapadá jen jako vedlejší efekt, ne jako služba, kterou bys měl považovat za svou zálohovací politiku. Common Crawl udržuje otevřený korpus webových crawlů, pracuje s WARC/WET daty, uvádí přes 300 miliard stránek napříč 15 lety a přidává 3–5 miliard nových stránek měsíčně. Pokud se tvoje živá zrcadla stanou crawlable a dostanou se do veřejného link graphu, můžeš skončit i tady. Ale spoléhat na to jako na plán by bylo stejně naivní jako spoléhat na náhodný leak URL. To je bonusový sediment, ne strategie. citeturn7view3turn7view4

**IPFS** patří do stejné kategorie „použij jako přídavek, ne jako hlavní nosník“. Jeho výhoda je obsahové adresování a to, že i při výpadku konkrétní HTTP gateway lze obsah získat, pokud ho stále poskytuje aspoň jeden uzel. Problém je z druhé strany stejně tvrdý: IPFS uzly zacházejí s daty jako s cache a bez pinningu není žádná záruka, že data zůstanou uložená. Oficiální dokumentace to říká napřímo. Proto má IPFS smysl jen tehdy, když CID zapíšeš do ostatních mirrorů a zároveň obsah připneš na více nezávislých místech nebo v clusteru. Bez toho je „decentralizace“ jen marketingový deodorant na nestabilní dostupnost. citeturn8view0turn8view1turn8view2

## Co je slabý nápad

Dropbox public link je pro tvůj use case špatný kandidát. Ne protože by technicky neuměl sdílet soubory, ale protože jeho failure mode míří přímo na tvůj scénář „nahraju a rok se nestarám“. Dropbox Basic podle své nápovědy čas od času maže neaktivní účty; za neaktivitu považuje 12 měsíců bez loginu nebo file activity, pouhé sdílení se nepočítá a po uzavření účtu budou zbývající soubory smazány. To znamená, že přesně ta „pasivita“, kterou chceš, je u Dropboxu sama o sobě rizikový trigger. Jako krátkodobý transit možná. Jako archivní sink ne. citeturn6view5

Podobně přeceňovaná je samotná Wayback Machine bez dalších vrstev. Save Page Now sice umí uložit konkrétní stránku včetně obrázků a CSS, ale neupraví z tebe zázračně archiv celého webu: neukládá celou strukturu, nespouští crawl celého webu a samo o sobě nezajišťuje budoucí aktualizace. Kdo udělá jen tohle, ve skutečnosti nearchivuje web, ale pár osamělých renderů. citeturn9view1turn6view6

## Jak poskládat vrstvu přežití tak, aby to nebyla fraška

Pro tebe dává smysl rozdělit publikaci na dvě linky. První linka je **zdrojová**: veřejný GitHub repo jako canonical origin, zrcadla na GitLabu a Codebergu, a po každém významném release nebo aspoň jednou ročně push do Software Heritage. Druhá linka je **artefaktová**: z přenosného buildu udělat archiv typu `site-offline-YYYY-MM-DD.zip` a ten samý balík poslat minimálně na GitHub Releases, Zenodo a Internet Archive item. Tím odstraníš třídní chybu, kdy se lidé snaží cpát všechno všude stejným způsobem. Zdroj chce forge a archiv zdrojů. Hotový web chce release asset a veřejný item. citeturn6view1turn6view3turn8view4turn9view0

Každý snapshot by neměl být jen ZIP. Měl by obsahovat i stroze praktické soubory: `README-RECOVERY.md`, `MANIFEST.txt`, `SHA256SUMS`, seznam mirrorů a pokud použiješ IPFS tak i CID. Tohle není byrokracie, ale obrana proti tiché smrti. Když se za deset let najde jen jeden artefakt bez kontextu, checksumů a vysvětlení, našel se v zásadě jen kus dat. Když se najde artefakt s manifestem, mirror listem a instrukcí k obnově, našla se rekonstruovatelná publikace. Pro balení většího množství souborů ostatně i Zenodo samo doporučuje ZIP archiv. citeturn8view4

Pro dohledatelnost pak potřebuješ minimálně dvě veřejná živá zrcadla na cizích doménách služby, ne jen na vlastní doméně. Důvod je brutalisticky jednoduchý: když ti spadne vlastní doména, nechceš přijít i o všechny vstupní body. GitLab umí veřejné HTTPS stránky na `gitlab.io`, Codeberg na `codeberg.page`, a oba systémy jsou přímo určené pro statické weby. Na tyto mirror landing pages pak dává smysl ručně pustit Wayback „Save Page Now“, ale jen pro klíčové vstupní stránky: homepage, sitemapu, rozcestník mirrorů a třeba hlavní sekce. Ukládat tím jednotlivě stovky stránek je ruční otročina s mizerným poměrem výkon/cena. citeturn7view1turn7view0turn9view1

Vedle toho dává silný smysl nominace do Webarchivu. Tvoje doména, jazyk a autorství mají českou vazbu a Webarchiv je nastavený právě na dlouhodobou archivaci českých online zdrojů. To je jedna z mála cest, jak z tvého osobního webu udělat alespoň potenciálně něco bližšího kulturnímu artefaktu než jen osobnímu hostingu. Nevýhoda je, že to není instantní self-service dump celého buildu; výhoda je, že to není jen další účet na další platformě, ale institucionální vrstva paměťové infrastruktury. citeturn6view4turn9view2

## Doporučené portfolio pro tvůj konkrétní případ

Kdybych to měl zredukovat na jednu realistickou, nepřikrášlenou strategii, vypadala by takto.

GitHub nech jako hlavní veřejný repozitář se zdrojáky a releasy. Totéž repo mirroruj na GitLab a Codeberg. Zdrojovou větev a tagované releasy pravidelně posílej do Software Heritage. Přenositelný offline build bal jednou za release nebo jednou ročně do ZIPu a tentýž soubor ukládej jako GitHub release asset, jako záznam na Zenodu a jako item na Internet Archive. K tomu provozuj alespoň dvě živá statická zrcadla kvůli crawlům a odkazovému grafu, a hlavní doménu plus jedno mirror URL nominuj do Webarchivu. To je portfolio, které rozděluje riziko mezi komerční forge, otevřený archiv zdrojáků, repozitář výzkumných artefaktů, veřejný archiv souborů a národní paměťovou instituci. Žádná jedna z těch vrstev není dokonalá. Právě proto fungují společně. citeturn6view1turn7view1turn7view0turn6view3turn8view4turn9view0turn6view4

Nejslabší kus tvého původního nápadu je Dropbox. Nejoriginálnější, ale zároveň reálně použitelný kus je kombinace Software Heritage pro zdroj, Zenodo + Internet Archive pro stažitelné snapshoty a Webarchiv pro institucionální českou archivaci. To je mnohem lepší než „jestli URL někam leakne, tak třeba přežije“. Leak není strategie. Redundance, různé třídy archivů a záměrně budovaný link graph strategie jsou. citeturn6view5turn6view3turn8view4turn6view4turn6view6
